

## Overview

The reference model represents the expected behavior of the Dual Port RAM.

Instead of relying on the DUT to determine the correct output, the reference model maintains its own software representation of memory. This software model predicts what the DUT should return for every read operation.

The scoreboard later compares these predicted values with the actual values produced by the DUT.

---

# Why a Reference Model?

A reference model acts as the "golden" implementation of the design.

Rather than comparing signals directly, the verification environment compares:

* Expected data generated by the reference model
* Actual data generated by the DUT

This approach is more scalable and closely resembles industrial verification methodologies.

---

# Internal Memory

The reference model stores memory contents using an associative array.

```systemverilog
int mem[int];
```

Each entry in the associative array represents one memory location.

Unlike a fixed-size array, memory locations are created only when data is written to them.

Advantages include:

* Efficient memory usage
* Automatic creation of entries
* Easy existence checking using `exists()`

---

# Communication Flow

```text
                Write Monitor
                     │
                     ▼
             Reference Model
              ▲            │
              │            ▼
        Read Monitor   Expected Data
                           │
                           ▼
                     Scoreboard
```

The reference model receives both write and read transactions.

Write transactions update the internal memory model.

Read transactions generate the expected output data.

---

# Processing Write Transactions

When a write transaction arrives, the reference model updates its memory.

```systemverilog
mem[write_addr] = write_data;
```

This mirrors the operation performed by the DUT.

---

# Processing Read Transactions

When a read transaction arrives, the reference model checks whether the requested address exists.

```systemverilog
if(mem.exists(read_addr))
```

If the address has previously been written, the corresponding data becomes the expected read value.

If the address has never been written, the reference model reports that no valid data exists for that location.

---

# Parallel Processing

The reference model uses parallel execution.

```systemverilog
fork
...
join
```

This allows write transactions and read transactions to be processed independently, reflecting the behavior of the dual-port RAM where reads and writes can occur concurrently.

---

# Expected Data Generation

Once the expected read value has been determined, the reference model packages the information into a transaction and forwards it to the scoreboard.

```text
Observed Read
      │
      ▼
Reference Model
      │
      ▼
Expected Read Data
      │
      ▼
Scoreboard
```

The scoreboard then compares this expected result against the actual output observed from the DUT.

---

# Advantages

* Independent software representation of memory
* Golden reference for verification
* Efficient associative-array implementation
* Supports concurrent read and write processing
* Produces expected results without relying on DUT outputs

---

# Summary

The reference model maintains an independent software representation of the RAM and predicts the expected output for every read request. By comparing these predictions with the DUT's actual responses, the verification environment can accurately determine whether the design behaves correctly.



## Overview

The transaction class represents a single memory operation in the verification testbench. Instead of manipulating interface signals directly, the verification environment exchanges transaction objects between components such as the sequence, driver, monitor, and scoreboard.

Each transaction contains all the information required to perform or observe a RAM operation.

---

# Why Use Transactions?

A transaction models an operation at a higher level of abstraction.

Instead of thinking in terms of individual signals, the verification environment works with complete memory operations.

For example, a write transaction contains:

* write enable
* write address
* write data

A read transaction contains:

* read enable
* read address
* returned read data

This abstraction makes the verification environment easier to understand, reuse, and extend.

---

# Transaction Fields

## Write Operation

| Field        | Description               |
| ------------ | ------------------------- |
| `write_en`   | Enables a write operation |
| `write_addr` | Memory address to write   |
| `write_data` | Data to be stored         |

---

## Read Operation

| Field       | Description              |
| ----------- | ------------------------ |
| `read_en`   | Enables a read operation |
| `read_addr` | Memory address to read   |
| `read_data` | Data returned by the DUT |

The `read_data` field is populated by the monitor after the DUT completes the read operation.

---

# Randomization

The transaction uses constrained random stimulus to generate a wide variety of memory accesses automatically.

Randomization allows the verification environment to exercise many different scenarios without manually creating every test case.

---

# Constraints

## Enable Constraint

```systemverilog
{read_en, write_en} != 2'b00;
```

This constraint prevents idle transactions.

Every generated transaction performs at least one valid memory operation.

---

## Address Constraint

```systemverilog
write_addr != read_addr;
```

This avoids simultaneous read and write operations targeting the same memory location.

This simplifies initial verification. More advanced tests can intentionally remove this restriction to verify read-during-write behavior.

---

## Data Constraint

```systemverilog
write_data inside {[1:400]};
```

The generated write data is limited to values between 1 and 400.

Restricting the range makes simulation waveforms easier to inspect during early development. The range can later be expanded to verify the full data width.

---

# Transaction Flow

```text
Sequence
      │
      ▼
Random Transaction
      │
      ▼
Driver
      │
      ▼
RAM
      │
      ▼
Monitor
      │
      ▼
Scoreboard
```

A transaction is first generated by the sequence, translated into signal-level activity by the driver, observed by the monitor, and finally checked by the scoreboard to verify correct DUT behavior.

---

# Summary

The transaction class forms the foundation of the UVM verification environment by encapsulating all information needed for a RAM operation. Using constrained random transactions improves stimulus generation, increases functional coverage, and keeps the verification environment modular and reusable.

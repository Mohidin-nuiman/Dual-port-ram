

## Overview

The read driver converts read transactions into signal-level operations that request data from the Dual Port RAM.

Unlike the write driver, it does not provide data to the DUT. Instead, it requests data by driving the read control signals, while the DUT generates the read data.

---

# Purpose

The read driver is responsible for:

* Receiving read transactions from the generator
* Driving the read enable signal
* Driving the read address
* Synchronizing all read requests with the system clock

---

# Communication Flow

```text
Generator
      │
      ▼
Mailbox
      │
      ▼
Read Driver
      │
      ▼
Interface
      │
      ▼
Dual Port RAM
      │
      ▼
Read Data
```

The read driver initiates memory read requests, and the DUT responds with the requested data.

---

# Receiving Transactions

The driver continuously waits for transactions.

```systemverilog
mbx1.get(t1);
```

Each received transaction contains the address that should be read from memory.

---

# Virtual Interface

```systemverilog
virtual ram_if.read_drv vif;
```

The virtual interface connects the driver to the DUT signals through the interface, allowing the driver to remain independent of the DUT hierarchy.

---

# Driving Signals

The read driver controls only the read request signals.

| Signal      | Purpose                                 |
| ----------- | --------------------------------------- |
| `read_en`   | Enables a read operation                |
| `read_addr` | Specifies the memory location to access |

The `read_data` signal is **not driven** by the driver because it is generated by the DUT during the read operation.

---

# Synchronization

All interface activity is synchronized using the read driver clocking block.

```systemverilog
@(vif.r_drv_cb)
```

Clocking blocks help eliminate race conditions between the verification environment and the DUT by defining when signals are driven and sampled.

---

# Read Sequence

For every transaction, the read driver performs the following operations:

```text
Receive Transaction
        │
        ▼
Assert read_en
        │
        ▼
Wait for clock cycles
        │
        ▼
Drive read_addr
        │
        ▼
Wait for clock cycles
        │
        ▼
Deassert read_en
```

Once the read request reaches the DUT, the memory returns the requested data on the `read_data` output, which is later captured by the monitor.

---

# Advantages

* Independent from the write driver
* Modular and reusable implementation
* Clock-synchronized signal driving
* Clear separation between stimulus generation and DUT response

---

# Summary

The read driver translates high-level read transactions into synchronized interface activity. It drives the read enable and read address signals while allowing the DUT to generate the corresponding read data. This separation of responsibilities results in a clean and modular verification architecture.

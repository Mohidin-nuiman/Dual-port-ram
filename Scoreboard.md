

## Overview

The scoreboard is the final checking component in the verification environment.

Its responsibility is to compare the actual data returned by the Design Under Test (DUT) with the expected data generated by the reference model.

If both values match, the RAM behaves correctly for that transaction. Otherwise, the scoreboard reports a verification failure.

---

# Purpose

The scoreboard performs the following tasks:

* Receives the actual read transaction from the read monitor
* Receives the expected read transaction from the reference model
* Compares the read addresses
* Compares the read data
* Reports pass or fail status

---

# Communication Flow

```text
             Read Monitor
                   │
                   ▼
            Actual Transaction
                   │
                   ▼
               Scoreboard
                   ▲
                   │
          Expected Transaction
                   ▲
                   │
           Reference Model
```

The scoreboard is the point where verification results are determined.

---

# Receiving Transactions

The scoreboard waits until both transactions become available.

```systemverilog
fork
    mbx1.get(...)
    mbx2.get(...)
join
```

The first mailbox provides the transaction observed from the DUT.

The second mailbox provides the expected transaction generated by the reference model.

Only after both transactions are available does the comparison begin.

---

# Address Comparison

The first verification step checks that both transactions refer to the same memory location.

```systemverilog
if(actual.read_addr != expected.read_addr)
```

A mismatch indicates that the transactions are not synchronized correctly or that an unexpected memory location was accessed.

---

# Data Comparison

If the addresses match, the scoreboard compares the returned data.

```systemverilog
if(actual.read_data != expected.read_data)
```

If the values differ, the DUT has produced an incorrect result.

Otherwise, the transaction passes verification.

---

# Verification Decision

The scoreboard produces one of three outcomes:

| Condition              | Result             |
| ---------------------- | ------------------ |
| Address mismatch       | Error reported     |
| Data mismatch          | Error reported     |
| Address and data match | Transaction passes |

---

# Overall Verification Flow

```text
Generator
      │
      ▼
Drivers
      │
      ▼
Dual Port RAM
      │
      ▼
Monitors
      │
      ├──────────────► Actual Transaction
      │
      ▼
Reference Model
      │
      ▼
Expected Transaction
      │
      ▼
Scoreboard
      │
      ▼
PASS / FAIL
```

The scoreboard does not generate stimulus or predict behavior. Its sole purpose is to determine whether the DUT output matches the expected result.

---

# Advantages

* Independent verification component
* Compares expected and actual results
* Detects address and data mismatches
* Provides clear pass/fail feedback
* Keeps checking logic separate from stimulus generation

---

# Summary

The scoreboard is the final verification stage of the environment. By comparing the DUT's observed output with the expected output from the reference model, it determines whether each read transaction is correct. This separation of prediction and comparison makes the verification environment modular, maintainable, and easy to extend.


## Overview

The interface acts as the communication bridge between the Design Under Test (DUT) and the verification environment.

Instead of connecting every signal individually to each verification component, the interface groups all DUT signals into a single reusable object. This improves readability, simplifies connectivity, and allows different verification components to access only the signals they require.

---

# Why Use an Interface?

Without an interface:

```
Driver
 ├── clk
 ├── write_en
 ├── write_addr
 ├── write_data
 ├── read_en
 ├── read_addr
 └── ...
```

Every component requires numerous signal connections.

With an interface:

```
Driver
      │
      ▼
+---------------+
|   ram_if      |
+---------------+
      │
      ▼
      DUT
```

Only one interface handle is passed to each verification component, making the testbench cleaner and easier to maintain.

---

# Interface Declaration

```systemverilog
interface ram_if(input bit clk);
```

The interface receives the system clock from the top-level testbench. All clocking blocks use this clock to synchronize signal sampling and driving.

---

# Interface Signals

## Write Signals

| Signal     | Description                        |
| ---------- | ---------------------------------- |
| write_en   | Enables a write operation          |
| write_addr | Address where data will be written |
| write_data | Data sent to the RAM               |

---

## Read Signals

| Signal    | Description              |
| --------- | ------------------------ |
| read_en   | Enables a read operation |
| read_addr | Address to be read       |
| read_data | Data returned by the RAM |

---

# Clocking Blocks

Clocking blocks synchronize signal access with the DUT clock.

They eliminate race conditions between the DUT and verification components by defining when signals are driven and when they are sampled.

Each clocking block operates on the positive edge of the clock.

---

## Write Driver Clocking Block

```systemverilog
clocking w_drv_cb @(posedge clk);
```

Used exclusively by the write driver.

Responsibilities:

* Drive `write_en`
* Drive `write_addr`
* Drive `write_data`

The driver controls the write interface of the RAM through this clocking block.

---

## Read Driver Clocking Block

```systemverilog
clocking r_drv_cb @(posedge clk);
```

Used by the read driver.

Responsibilities:

* Drive `read_en`
* Drive `read_addr`

The driver requests data from the DUT through these signals, while `read_data` is sampled as an input because it is generated by the DUT.

---

## Write Monitor Clocking Block

```systemverilog
clocking w_mon_cb @(posedge clk);
```

The write monitor observes write transactions without affecting the DUT.

It samples:

* write enable
* write address
* write data

These observations are converted into transactions and forwarded to the scoreboard.

---

## Read Monitor Clocking Block

```systemverilog
clocking r_mon_cb @(posedge clk);
```

The read monitor observes:

* read enable
* read address
* read data

It captures completed read operations and sends them to the scoreboard for comparison.

---

# Timing Control

Each clocking block uses

```systemverilog
default input #1 output #1;
```

This specifies a one-time-unit skew for both inputs and outputs.

Benefits include:

* avoiding race conditions
* stable sampling of DUT outputs
* predictable timing between the DUT and UVM components

---

# Modports

Modports restrict how each verification component accesses the interface.

Instead of exposing every signal to every component, each modport provides only the required clocking block.

The interface defines four modports:

| Modport     | Used By       |
| ----------- | ------------- |
| `write_drv` | Write Driver  |
| `read_drv`  | Read Driver   |
| `write_mon` | Write Monitor |
| `read_mon`  | Read Monitor  |

This separation improves encapsulation and prevents accidental access to unrelated signals.

# Data Flow

```
Sequence
     │
     ▼
Driver
     │
     ▼
Interface
     │
     ▼
Dual Port RAM
     │
     ▼
Interface
     │
     ▼
Monitor
     │
     ▼
Scoreboard
```

The interface serves as the central communication channel between the DUT and all verification components, ensuring synchronized, race-free signal transfers.

---

# Summary

The `ram_if` interface encapsulates all RAM signals, provides synchronized access through dedicated clocking blocks, and controls signal visibility using modports. This organization makes the verification environment modular, scalable, and easier to maintain as additional features or tests are added.
